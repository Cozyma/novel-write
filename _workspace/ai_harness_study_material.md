# AIハーネス設計の教訓 — 全ゲート緑で合格率1.5%だった話

システム開発におけるAIハーネス（AIエージェントを組み込んだ開発パイプライン）設計の勉強会資料。
題材は変わり種——**AIと人間の二人チームで連載小説を書くリポジトリ**——だが、起きた失敗と効いた対策はすべてソフトウェア開発に写像できる。2026-07-06時点の実ログに基づく。

---

## 0. 題材：小説リポジトリを「ソフトウェア開発」として運用した

このプロジェクトは、小説制作にソフトウェア開発のプラクティスを全面移植している：

| 移植したプラクティス | 小説側の実装 |
|---|---|
| SSOT（単一情報源）・正本→派生同期 | 章サマリー＝正本、索引CSV・上位サマリー＝派生。片側更新禁止 |
| スキーマ＋CI | YAMLスキーマ＋自作バリデータ（負荷上限・必須キー・生成ゲート） |
| pre-commit フック | 廃止語彙の残骸grep＋台帳未反映チェック |
| 中間表現（IR） | event_timeline＝描写に依存しない事象列（WHATのみ） |
| リグレッションテスト | ブラインド再生成実験（コールドなAIに同じ入力から再生成させ、差分を測る） |
| フレッシュアイレビュー | コールド通読（文脈を持たないエージェントによる指摘専用パス） |
| 受け入れテスト | 人間（作者）の読了判定 |

特徴は、最重要品質「**面白いか**」が機械観測不能なこと。ソフトでいう「ユーザー価値」「使い心地」「保守性」のような非機能価値が主戦場であるため、**AIハーネスの構造的な失敗が濃縮された純粋な形で現れる**。

## 1. 事件：オール緑で合格率1.5%

パイプラインは多層のゲートを持っていた——各階層に「面白さの核」の必須フィールドと検査、スキーマ検査、認知負荷の上限、AIセルフレビュー、執筆着手前の核ゲート。4話分の本文——評価のため場面・出来事の最小単位に分解すると65項目——は**全ゲートを通過して**生成され、多重の改稿を経ていた。

作者が初めて全65項目を実測評価した結果：**◎（本当に面白い）は1つ**。

ソフトに翻訳すると：CI全緑・レビュー承認済み・Definition of Done 充足でリリースした65機能のうち、顧客が価値を感じたのは1個。

この事故を分解すると、3つの設計エラーが出てくる。

## 2. 教訓A：品質は定式生成できない——類型はリンタであってコンパイラではない

**何をしていたか**: 「面白さ＝7変数（探求・報酬・驚き・危機…）の刺激の類型」というモデルを立て、変数のピークを章に配置すれば面白さが生成できる前提でパイプラインを組んでいた。

**何が起きたか**: 観測された「良いもの」を規範化するたびに出力が平板化した（3連敗：物語の中間点に「どんでん返し」を義務化→撤去／「毎回ギリギリの緊張」をノルマ化→戦闘場面の設計自由度を縛って撤去／実測で唯一の最高評価だった場面型「誤解した実力者が、勝手に主人公の評価を上げてくれる場面」を毎話義務化しかけて疑義で停止）。そして最高評価（◎）の場面と不満（×）の場面は**同じ類型ラベルを持っていた**——効いていたのは類型ではなく、個別の布置（具体性・文脈・タイミング・希少性）だった。

**ソフト開発での同型**:
- コードカバレッジを目標値にする → 意味のないテストが量産される（Goodhartの法則）
- デザインパターンをレシピとして適用 → 過剰設計（パターンは本来、良い設計の事後の診断語彙）
- 「Netflixがやってるから」のベストプラクティス移植 → 文脈を剥がした類型だけのカーゴカルト

**AIハーネスへの示唆**:
- プロンプトに「良い出力の特徴リスト」を積み上げるほど、**チェックリスト準拠の平板な出力に収束する**。良さは特徴リストでなく**承認済み実例（exemplar・ゴールデンサンプル）のコーパス**で与える。
- チェックリスト・メトリクスは**欠乏と過剰の検出（リンタ）に限定**する——「読者への見返り（カタルシス）が無い章が5連続」「一文の長さ・新出用語密度の超過」のような確実な欠陥検出には安定して効いた。生成の目標関数にした瞬間に腐る。

## 3. 教訓B：観測ゲートと論証ゲート——LLMは論証ゲートの最強通過者

**何をしていたか**: 執筆着手前のゲートに「この章の面白さの核は成立しているか」という基準を置いていた。

**問題の構造**: この基準は成果物を読む前には**観測できない**。観測できない基準の通過判定は、必然的に**論証**（それらしい理由付け）で行われる。そして生成AIは「もっともらしい論証」の専門家である——**論証で通れるゲートは、AIに対して情報量ゼロになる**。

**何が起きたか**: 偽陽性の量産（ゲート通過65ビート中◎1）。もう一つの症状が**設計メモの弁明肥大**——あるシーンは局所補修を3回重ね、そのたびに「なぜこれで良いか」の注釈が育った（最終的に構造ごと差し替えて解決。補修2回で直らないのは補修対象が間違っているシグナル、という教訓も併せて得た）。

**ソフト開発での同型**:
- ウォーターフォールの Big Design Up Front：設計書レビューのゲートは設計書という論証の出来を選抜するだけだった。アジャイル宣言の「包括的なドキュメントよりも動くソフトウェアを」は「論証ゲートを観測に置き換えろ」の一文
- セキュリティ監査チェックリスト通過（論証） vs ペネトレーションテスト（観測）
- 「この設計は保守性が高いか？」をレビュー項目にする → レビューが修辞の勝負になる

**AIハーネスへの示唆**:
- **ゲートに置いてよいのは機械検査・実行で判定できる基準だけ**：型・テスト・スキーマ・lint・実行結果・grep。
- AIのセルフレビュー・self-critique ループ、LLM-as-judge も**論証側**にいる。補助には使えるが、ゲート（通過判定の権威）にすると同じ穴に落ちる。
- 観測不能な価値の検査は廃止でなく**「仮説の宣言」に降格**する——「この章の核はこれだ」と書かせる効用（具体化の強制・空白の検出）は残るが、**通過は品質を保証しない**と位置づけを明文化する。

## 4. 教訓C：オラクルを飢えさせるな——AIは「まぁいいか」を忠実に増幅する

**何をしていたか**: 唯一の実観測器＝オラクル（品質の最終判定者。ここでは作者の読了判定、ソフトなら実ユーザー）をパイプラインの最後尾に置いていた。上流のゲートが緑を出し続けるので観測は先送りされ、正式判定済みの成果物が**ゼロ**のまま4話分の量産と多重改稿が進んだ。

**AIが加害者に見えて違う点**: 事後分析の結論は「**AIは『まぁいいか』採用の正本を死守条件として扱い、その上に増築する失敗を反復した**」。人間の部下なら曖昧な仕様を暗黙に直すところを、AIは正本への忠実度が高いため、**未検証の暫定品質をそのまま高精度に増幅する**。しかも生成が安いので、未検証在庫の積み上がる速度は人間チームの比ではない。根因はAIでなく**観測の飢え**である。

**ソフト開発での同型**:
- 社内プロキシ（仕様承認・ステークホルダーサインオフ・QA通過）が緑のまま長期開発→ビッグバンリリースで初観測→爆死。リーンスタートアップ（build-measure-learn）はこの失敗への対策として発明された
- 「プロファイリングせずに最適化」——測定より推論を信じる同型

**AIハーネスへの示唆**:
- **観測は早く・小さく・頻繁に**。量産（詳細設計・一括生成）の前に、価値仮説の最小試作をオラクルに当てる工程を挟む（このリポは再構成にあたり「章構成の詳細設計の前に、最高評価が狙える場面の短い試作を作者判定に当てる」工程をパイプラインに挿入した）。
- プロキシの緑を「観測不要」の根拠にしない。
- オラクルの判定（◎/×）は**蓄積してコーパス化**し、教訓Aの exemplar として次の設計の出発点にする。

## 5. 効いたハーネス機構（実測済みのもの）

失敗だけでなく、実験で効果が確認された機構がある。ソフト開発への対応物と併記する。

| 機構 | 内容 | ソフト側の対応物 | 実証 |
|---|---|---|---|
| **decision_log** | AIが正本にない判断（解釈・補完・発明）をした場合、成果物と必ずセットで提出させる | AIエージェントのPR説明・仮定の明示 | 暗黙に埋まった判断は「それらしく読めてしまう」ため検出不能になる——可視化で人間レビューが機能した |
| **中間表現の分離** | event_timeline（WHATのみの事象列）を正本とし、描写（HOW）と分離 | 仕様とレンダリングの分離・IR | 語り口・視点を変えた再生成で事象が保存される。物理整合の検算も描写を盛る前の事象レベルで可能に |
| **ブラインド再現性テスト** | コールドなAIに正本群だけ渡して同じシーンを再生成させ、正本とのズレを測る | ゴールデンテスト・クリーンルーム再現 | 人間フィードバックの正本への書き戻しが「システムに固定された」ことを実測（書き戻し済みのルールは再発ゼロ）。ズレは**正本が曖昧な箇所に正確に集中**——「会話で合意しただけのものは再現されない。正本に書かれたものだけが再現される」 |
| **コールド通読** | 文脈を持たないエージェントに最小参照＋本文だけ読ませ、**指摘のみ**させる（修正させない・1パス限り） | フレッシュアイのコードレビュー | 7巡の改稿で人間・AI双方が見逃した時系列バグを検出。「温まった文脈は生成には毒、検出には資産」 |
| **人間判定をループ終了条件に** | 自動修正ループを回しきらず、終了判定を人間に固定 | 受け入れテスト・UAT | 自動ループだけ回すと「全チェック通過の平板な出力」に収束する（教訓Bの帰結） |
| **tripwire（死語台帳）** | 廃止した概念・語彙をgrep可能なパターンで台帳化し、pre-commitで機械検査 | deprecated API の lint ルール | 設計変更の残骸検出が目視頼みだった状態を機械化。「登録がないと後から残骸検査を打てない」 |

## 6. なぜAIで失敗が加速するのか（4つの構造要因）

1. **LLMは論証の専門家**——論証で通れるゲートは必ず突破される。ゲートの選抜対象が「価値」から「もっともらしさ」にすり替わる。
2. **LLMは忠実**——未検証の「まぁいいか」正本も、人間のフィードバックも、同じ精度で増幅する。前者を放置すると誤差が複利で増える（忠実さは還流が回っていれば最強の資産になる。§5のブラインドテスト参照）。
3. **生成が安い**——未検証在庫の積み上がる速度が人間チームの比でなく、オラクルの処理能力との差が開き続ける。
4. **自動ループはGoodhartを自動化する**——self-fix ループは検証項目への過適合を高速で完成させる。

## 7. 持ち帰りチェックリスト

自分のAIハーネスに対して：

- [ ] そのゲート、**観測か論証か**？（実行・機械検査で判定できるか、それらしい文章で通ってしまうか）
- [ ] **オラクル**（人間・実ユーザー）はパイプラインのどこにいるか？ 最後尾に一つだけなら、飢えている。
- [ ] AIに増幅させている**正本は検証済みか**？ 「暫定・まぁいいか」のまま下流に流していないか。
- [ ] 良さを**特徴リスト**で教えていないか？ **承認済み実例**で教えているか。
- [ ] AIの**暗黙判断は可視化**されているか（decision_log 相当があるか）。
- [ ] 人間フィードバックは**会話の中で消えていないか**——正本ドキュメントへ書き戻され、再現テストで「固定された」ことを実測できるか。
