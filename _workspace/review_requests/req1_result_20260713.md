検査モードで実施しました。読んだのは依頼パケット・WORK_MASTER.md全文・ARCHITECTURE.md §3節・TODO.md冒頭のみです（TODOは読み込み単位の都合で§0より下も画面に入りましたが、指摘の根拠は§0までに限定しています）。指摘のみ、以下16件。

Q1 単一書き手の穴
1.【高】 引用: 鉄則①「作話・検査セッションは**§4起票箱・§5受信箱への追記だけ**」＋§4見出し「起票箱（ルール変更案・追記のみ）」＋§5「セッション終了報告・追記のみ」
根拠: ルール変更案でも終了報告でもない中間発見（他話の整合バグ・状態変化の気づき）の受け皿が未定義で、書き場所に困ったセッションが§3や CONTEXT へ直接書く抜け道になる。
改善案: §5を「終了報告＋随時の発見メモ」に拡張するか、発見箱を1節新設。

2.【高】 引用: ARCHITECTURE [Hook: コミット前]「未反映があればコミット前に書き込む——作業状態領域 _workspace/CONTEXT.md _workspace/TODO.md」
根拠: コミット前フックがマスタ導入前の文面のままで WORK_MASTER の扱いを定めておらず、「未反映を埋めよ」に忠実なセッションほど§3トラック表を直接更新してしまう。
改善案: 同フックに「WORK_MASTER の反映は管理のみ・他モードは§5へ追記」と1行追加。

3.【中】 引用: §1検査「コールド1体への委任可（親は結果の記帳のみ）」
根拠: 「記帳」の書き先が未指定で、§3トラック表への直接記入（＝本体書き換え）とも読める。
改善案: 「記帳＝§5への追記」と明示。

4.【中】 引用: 鉄則④「100行以内を維持」
根拠: 行数を超過させ得るのは追記する作話・検査だが、是正（圧縮＝本体書き換え）は管理のみで、超過時にその場で「善意の圧縮」をする誘因が構造的に生まれる。
改善案: 「超過しても放置し§5で報告・圧縮は管理」と明記。

5.【低】 引用: §1「開始時に1つ宣言し、CONTEXTの着手宣言に併記」
根拠: モードは自己申告のみで、宣言と実作業の一致を検査する仕組みがなく、宣言だけ作話で実質マスタ改変をしても検出されない。
改善案: pre-commit フックで「WORK_MASTER の§4/§5以外の差分 × CONTEXT の宣言モード」の突き合わせを非ブロック表示。

Q2 マスタ自身のドリフト経路
6.【高】 引用: ARCHITECTURE [Hook: セッション終了・タスク質変化時]（内容＝リセット提案と草案永続化のみ）
根拠: §5の報告書式は定義されているのに、終了フック側に「§5へ報告を追記する」義務が配線されておらず、報告漏れ→受信箱が空→管理が消化する材料なし→§3が静かに乖離する、が最もありそうな筋。
改善案: 終了フックに「§5へ終了報告を追記」を1行追加。

7.【高】 引用: §1管理「アジェンダ固定＝①マスタと実態の乖離チェック…」
根拠: 管理セッションの起動条件・頻度の定めがなく、更新責務が管理に集中する設計ゆえ、管理が開かれない期間はマスタ全体が凍結し、開始フックが全セッションに古い状態を復元し続ける。
改善案: 起動トリガ（例: §5に2件溜まったら次セッションは管理）を明文化。

8.【中】 引用: 鉄則②「再話禁止＝中身はポインタと状態のみ」vs §3セル「v1判定中断＝作者疑義「同伴と男が無意味」→v2再配線案（段5視線の対象＋段8内語＝割符の線の読み）提示済み」
根拠: 起草時点で既にセルが経緯の再話を含んでおり、消化のたびに散文が蓄積→100行制限と衝突→要約時の情報落ちか制限の形骸化、という筋が初日から実演されている。
改善案: セル書式を「状態語＋ポインタ（CONTEXT続n）」に固定。

9.【中】 引用: §5書式「…目標照合の自己申告」＋管理②「受信箱の消化→§3更新」
根拠: 診断が「ドリフト源＝自己認証」なのに、管理は生成側の自己申告をそのまま§3へ転記する設計で、自己認証がマスタ経由で公式記録化（ロンダリング）される。乖離チェック①の照合手続きも未定義。
改善案: 管理②に「申告と成果物パスの抜き取り照合」を1行加える。

Q3 モード境界の曖昧箇所
10.【高】 引用: §1作話「設計→構造承認→レンダー→⑦セルフレビュー…まで。ルールは読み取り専用」
根拠: 読み取り専用なのは「ルール」だけで、canon・骨格・前話・他話の scene_input という正本群の修正が作話に含まれるか未定義——レンダー中に他話由来の整合バグを見つけた場合、ARCHITECTURE ③''(a) の input 先行改稿に従ってその場で他話の正本まで直すのは作話の自然な読みであり、越境改稿の歯止めがない。
改善案: 「現タスク対象話の input/本文まで＝作話、他話・canon の修正＝起票して別途」と線引き。

11.【中】 引用: §1検査「コールド1体への委任可」vs ARCHITECTURE ④'「コールド通読（任意・人間判定の前哨）」
根拠: 作話ワークフロー内の④'と検査モードのコールド委任が同型で、④'を回した作話セッションが「検査済み」を自認する混同（検査分離の骨抜き）が起きやすい。
改善案: 「④'＝作話内の前哨・目標適合チェック＝別セッションの検査」と相互参照を明記。

12.【中】 引用: 管理④「直近提出物の検査手配」vs §1検査「依頼パケット…親が対象本文を抽出して同梱」
根拠: パケット作成（本文抽出・設問起草）が管理の「手配」か検査親の仕事か二重に読め、設問起草は生成寄りの作業のため管理の「作話を持ち込まない」とも擦れる。
改善案: 作成主体を一方に確定（例: 管理が作成・検査は実施のみ）。

13.【低】 引用: §1「モード外の作業が必要になったら、その場でやらず§4/§5に書いてセッションを切り替える」＋ARCHITECTURE [Hook: 決定事項発生・タスク完了時]「CONTEXTとTODOをセットで同期更新」
根拠: タスク完了という同一イベントの記録先が、TODO/CONTEXT更新（全モード可）とトラック表更新（管理のみ→§5経由）に分裂し、どこまでをその場で書いてよいか毎回迷う。
改善案: §1運用に「完了時＝TODO/CONTEXT更新＋§5報告・§3は触らない」の1行手順を置く。

Q4 発効手順の矛盾
14.【中】 引用: 冒頭「発効: 次セッションから」vs §5「※直近の消化: 2026-07-10依頼2報告→§3トラック表反映済み（管理 2026-07-10）」

改善案: 発効行を「起草日から有効」に改めるか、発効待ち項目だけを列挙。

改善案: 「§2/§3＝即時・§1/鉄則＝次セッション（改定行に日付印）」と定義。

改善案: どちらかに発効留保を揃える。
