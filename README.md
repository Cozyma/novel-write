# 小説執筆・自動更新フレームワーク

本リポジトリは、人間とAIエージェントが協働して小説を執筆・推敲し、関連するデータを自動更新するためのフレームワークです。

## 1. 全体ディレクトリ構造

```plaintext
/novel_framework_repo
├── README.md                 # 本指示書（システムルール・運用フローの定義）
├── _workspace/               # 【自由検討領域】人間とAIの壁打ち・メモ（AIは指示なしに読み込まない）
│   ├── 01_macro_notes/       # 企画・キャラクター設計の検討ログ
│   ├── 02_meso_notes/        # 話数ごとの展開や変数のピーク調整
│   └── 03_micro_notes/       # シーン内の具体的な行動（Fact）の推敲
├── 00_system/                # 【システムルール】プロンプトと制約条件の定義
│   ├── variable_rules.md     # 7変数の定義、および【負荷】制限などの判定基準
│   └── prompt_templates/     # 処理別（生成、推敲、状態抽出）のプロンプト
├── 01_static_database/       # 【不変データ】世界観・キャラクターの絶対的真実
│   ├── characters/           # 人物ファイル（経歴、口調、Wants / Needsの定義）
│   ├── world/                # 世界観、地理、用語集、歴史的ルール
│   └── timeline.md           # 絶対時間の総合年表（出来事が起きた時系列）

│   ├── character_states.yaml # 各キャラの現在地、負傷状況、疲労度
│   ├── resources_states.yaml # 物理アイテムの現在位置、所有者（チェーホフの銃）
│   └── reader_knowledge.yaml # 読者および視点人物の情報開示・誤認ステータス
├── 03_macro_concept/         # 【作品方針】作品全体のコンセプトと優先変数の定義

├── 05_micro_scene/           # 【執筆指示】AIへの直接的なインプット
│   └── episode_XX/
│       └── scene_YY_input.yaml# 起こる事実（Fact）と段落ごとの変数指定
├── 06_draft_output/          # 【AI生成結果】AIが出力した一次原稿

│       └── scene_YY_eval.json# テキストに対するAIの変数自己評価スコア
```

## 2. 7変数のスコープ（解像度）定義

AIエージェントは、指定された階層に応じて変数を厳密に評価・適用しなければならない。

### シーン単位で管理する変数（マクロ駆動系）
- **【探求】（情報ギャップ）**: シーン内で読者に提示する未解決の謎。
- **【報酬】（カタルシス）**: 謎の解明、目標の達成によるストレス解放度。
- **【驚き】（予測の裏切り）**: 読者の前提を覆す展開の有無。
- **【危機】（生存への脅威）**: タイムリミットや敵対者による状況の緊迫度。

### 段落単位で管理する変数（ミクロ処理系）
- **【負荷】（認知処理負荷）**: 一文の長さ、新出用語の密度、キャラ登場数（ブレーキ変数）。
- **【感覚】（身体的認知）**: 視覚・聴覚・触覚など五感に訴える語彙の量。
- **【共感】（心理的同期）**: 出来事に対するキャラクターの瞬間的な感情反応・内省の量。

## 3. シーン執筆・自動更新の実行フロー（Hook仕様）

AIエージェントは、新しいシーンを執筆する際、以下の5つのステップを自律的かつ連続的に実行する。

- **Step 1: Pre-Hook（前提条件の同期）**
  - **処理**: `05_micro_scene/` の指示ファイル、および `02_dynamic_states/` 内の最新ステータス（現在地、負傷、所持品、認知情報）をすべて読み込む。
  - **制約**: 状態データに存在しないアイテムを登場させたり、現在地と矛盾する描写をしたりすることを禁止する。
- **Step 2: Generation（描写のレンダリング）**
  - **処理**: 読み込んだ事実（Fact）をベースに、`04_meso_plot/` で指定されたシーンタイプ（Action/Reaction）の型、および段落ごとの変数パラメーターに従って本文を生成する。
  - **出力先**: `06_draft_output/episode_XX/scene_YY_draft.md`
- **Step 3: Post-Hook（状態差分の抽出）**
  - **処理**: 生成した本文を解析し、シーン内で発生した物理的・認知的な変化（移動、負傷、アイテム消費、新情報の獲得など）をJSON/YAML形式で抽出する。
- **Step 4: Commit（状態の機械的上書き）**
  - **処理**: Step 3で抽出した差分データを、`02_dynamic_states/` 内の各ファイル（character、resources、reader）に上書き保存し、次シーンへ状態を引き継ぐ。
- **Step 5: Validation-Hook（論理チェック）**
  - **処理**: 更新後の状態データが、`01_static_database/` の世界観ルールやタイムラインと矛盾していないかを機械的に検証する。【負荷】が上限値（80%）を超えている場合、または論理破綻を検知した場合は処理を停止し、人間にアラートを出力する。

## 4. エージェントへの基本命令（Prompt Grounding）

> 「あなたはこのリポジトリ内のデータを統制・出力するAIエージェントです。文章の生成、および状態の更新を行う際は、直感による記述を一切排除し、各ファイルに定義された客観的数値と事実（Fact）にのみ基づいてロジックを展開してください。人間が `05_micro_scene` に入力した指示が、本システムの唯一の起動トリガーとなります。」
