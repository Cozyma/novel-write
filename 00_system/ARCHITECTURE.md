# ARCHITECTURE.md — 4階層アーキテクチャとフック構造

## 1. 4階層アーキテクチャ（出力マッピング）
当プロジェクトは、エンタメ性と一貫性を担保するため以下の4階層で構成される。

- **L1: Macro Concept** → `03_macro_concept/` (作品全体の究極目標・危機)
- **L2: Meso Arc** → `04_meso_plot/arc_XX/` (カタルシス曲線を持つ中長期シナリオ・章)
- **L3: Micro Chapter** → `04_meso_plot/arc_XX/` 内の `chapter_summary.yaml` (連載1話分の変数刺激管理とミニ三幕)
- **L4: Nano Scene** → `05_micro_scene/` (AI執筆用のパラメーター指示)

## 2. 完全チェーン・フック構造（必須参照ルール）
下位構造のドキュメントやプロンプトを作成するAIエージェントは、作業前に必ず**直上の上位構造ドキュメント**を読み込むこと。

- **Arc作成時**: `03_macro_concept/template_concept.yaml` を参照。
- **Chapter作成時**: 上位の `04_meso_plot/arc_XX/arc_summary.yaml` と `chapter_sequence.csv` を参照。スキーマは `04_meso_plot/template_chapter_summary.yaml` に準拠（`catharsis_core` 必須・設計タグは `design_notes` へ分離。検証観点は `00_system/SELF_REVIEW.md` の【面白さの核の実在検証】）。
- **Scene作成時**: 上位の `04_meso_plot/arc_XX/chapter_YY_summary.yaml` を参照。着手前に `catharsis_core` が具体的な仮核として成立していることを確認（空白・約束手形なら執筆せず再設計へ差し戻し）。執筆中に出来事レベルの発明が生まれた場合は該当章の `catharsis_core` へ還流すること（`00_system/SELF_REVIEW.md`【面白さの核の実在検証】参照）。
- **全AI共通**: 変数を操作・マッピングする際は必ず `00_system/variable_rules.md` を参照すること。

### 2.1 データ権威・同期規則（正本と派生）
- 同一のビート・事実が複数ファイルに重複して存在する場合、`chapter_YY_summary.yaml` を**正本**とし、`chapter_sequence.csv`（索引）および `arc_summary.yaml`（acts の構造役割・reward_chain 等）は**派生**とみなす。
- 修正は必ず「正本→派生」の順で一括同期する。派生のみ・正本のみの片側更新を禁止する。
- **派生は構造のみ・出来事の再話を持たない**: 上位サマリーの派生表現は、構造役割・接続・番号台帳（章範囲・role・reward_chain 等）のみを持ち、下位（章）の出来事を言い換えた散文を持たない。章の事実を再話した派生散文は構造的に必ずドリフトする（2026-06-16に arc_summary の `episode_index` と `acts` 内の出来事再話〔hook/incident/showdown 等〕を廃止した実地証拠）。事実の正本は常に最下層（章サマリー／本文）に置き、上位は「どの単位が何の構造的役割を担うか」だけを指す。
- **語彙の単一出典**: 2ファイル以上で使用する役職・組織・固有名詞・システム用語は、必ず `01_static_database/` に定義を置いてから使用する。定義の無い名詞が複数ファイルに広がっている状態は「独り歩き」として検出・是正の対象とする。
- **本文と動的状態**: 執筆開始後、シーン内の事実の正本は**本文**（`06_draft_output/`→`07_final_text/`）である。`02_dynamic_states/` は「本文から再構成困難な状態」のみを持つ派生台帳——①長距離の事実（隣接コンテキスト外で回収される仕込み）、②累積カウンタ（全話横断の合計値）、③認識・誤読台帳（誰が何を知り、どう誤読しているか＋読者開示）。短距離の物理状態（現在地・負傷・疲労・所持品）は台帳管理せず、直近の本文・章サマリーから再構成する。更新は毎シーンではなく章完成・人間判定通過後（README §3参照）。

### 2.2 面白さの核の階層検査（従属チェーン）
- 従属関係: **7変数**（`00_system/variable_rules.md`）→ **L1コンセプト**（マクロ駆動系4変数〔探求/報酬/驚き/危機〕の抽象レベルでの集約＝catharsis_engine・priority_variables）→ **キャラDB**（`01_static_database/characters.yaml`。コンセプトに従属）→ **L2アーク／L3チャプター**（コンセプトとキャラの両方に従属）。
- 各レイヤーは、①そのスケールでの「面白さの核」フィールドと、②親レイヤーの核のインスタンスであることの検査を、必ず**対**で持つこと（検証可能性バイアスへの防御。SELF_REVIEW.md 7条参照）。
  - **L1**: `catharsis_engine`（面白さの公式）── 検査：本作の面白さがマクロ4変数の式として具体的に言えるか
  - **キャラ**: `story_hooks.concept_relation` ── 検査：このキャラがどの変数を物語のどの段でどう駆動するか一文で言えるか（SELF_REVIEW【キャラの変数駆動検査】）
  - **L2アーク**: `macro_linkage.sub_catharsis`（このアークの最大カタルシスの具体形）── 検査：catharsis_engineのインスタンスになっているか
  - **L3章**: `catharsis_core`（章の具体的な手）── 検査：アークの sub_catharsis へ寄与しているか

### 2.3 アーク構造のメソッド（カタルシス曲線＋最小借用）
アーク（約19章＝映画1本のスケール）の物語曲線は、**本作自身のカタルシス曲線（積み上げ→絶望→蹂躙＝setup→buildup→despair→catharsis、読みレンズは起承転結）**を背骨とする。段名『蹂躙』は**観測者から見たラベル**であり、実態は綱渡りの決着（読み・工夫・代償のやりくり。concept.yaml / catharsis_engine.priority_rule の2026-07-02再定義）——超火力の蹂躙は数アークに一度の縦の報酬軸の頂点で、章・アークの標準決着ではない。ハリウッド式脚本メソッドは「一回完結・映画1本」用に設計され連載と衝突しやすく（2026-06-09にL1固定で失敗→排除しすぎ→ハイブリッド、さらに2026-06-16にアークの4幕レンズ自体が注釈で打ち消される負債——midpoint=twist要件が危機ドリブンの本作で常に弱い——と判明し撤去、という経緯がある）、**規範ビート表としては持ち込まない**。借用するのは推進力を担保する2チェックのみ。

- **適用する層**:
  - **L2アーク層（主戦場）**: カタルシス曲線（4段）＋ drive_variable を `template_arc_summary.yaml` の `acts` で運用。ハリウッド式からの借用は2チェックのみ——**不可逆点**（buildup に後戻り不可の決定/事象。`has_irreversible_point`）と**能動的選択の着地**（catharsis の `embodiment_of_will`＝古い自分では打てない手。`has_embodiment_landing`）。midpoint=twist や all is lost 等の規範ビートは要件化せず、曲線の性格（危機/驚きドリブン）は drive_variable と design_notes で記述する。
  - **キャラ層（バディ二人制）**: キャラクターアークは2本柱で運用する——ノルク＝`concept.yaml` の `protagonist_growth_roadmap`、マーレン＝`characters.yaml` の `marlen.growth_roadmap`（ノルクと鏡映：打算の仮面／教条の鎧）。各アークの `roadmap_step` で**二人制検査**を行う：①各キャラ1アーク1段以内（早回し防止）②アークごとに主役サイドの誰か1人以上が前進（前進担保）③ノルクが2アーク連続で据え置きにならない（主人公の長期停滞検出）。**この成長設計は本作の最重要箇所のため両ロードマップを `status: provisional` とし、各アーク設計の着手時に「現在の段の確認」だけでなく「段定義そのものの妥当性」も必ず見直す（固定しない）。**
  - **L1（語彙と賭け金のみ）**: wrong_belief＝`internal_flaw`【誤闘】、evils.external＝`core_conflict`【究極の危機】のマッピングで取り込み済み。`stakes`（win/lose）は固定項目として有効。
  - **長距離台帳**: setup/payoff規律は `02_dynamic_states/resources_states.yaml`（planted/fired管理）で運用。
- **適用しない層（禁止）**:
  - **L1への一回性三幕の固定**: 連載は終わりが未定のため、作品全体に一回完結の三幕構造を張らない。Quest/Crisis等の展開項目はL1の固定項目にしない（章・アークから動的に生成される結果）。
  - **L3章へのビート表の強制**: 章はミニ三幕＋catharsis_coreで完結しており、save-the-cat型のビート表を章単位に重ねない。
  - **アークへのハリウッド式規範ビート表**: midpoint=驚きの最大化/twist・all is lost 等の一回完結曲線のビートをアークに強制しない。借用は不可逆点と能動的選択の着地の2チェックに限る（2026-06-16にレンズごと撤去した経緯。`arc_summary.yaml` / design_notes.curve_framing）。

## 3. イベント・運用フック（タイミング別ルール）
特定のタスクタイミング（イベント）が発生した際、AIエージェントは以下のフック処理を自律的に実行すること。

- **[Hook: セッション開始時]**
  - 新規セッション開始時、必ず `_workspace/CONTEXT.md`（直近のみ）と `_workspace/TODO.md` を読み込み、現在の作業文脈を復元する。
- **[Hook: タスク着手時]**
  - 作業を開始する前に、現在の取り扱いタスクを必ず `_workspace/CONTEXT.md` に追記（宣言）してから進める。
  - 読み込み前に `00_system/REFERENCE_MAP.md` を参照し、作業ケースに応じた**最小参照セット**を決めてから読む（トークン節約。全ファイルの無差別読み込みを避ける）。マップに無い作業ケースが発生した場合は、タスク完了時にマップへ実績を追記する。
- **[Hook: 決定事項発生・タスク完了時]**
  - 新たな決定事項や保留が生じた際は、`CONTEXT.md` と `TODO.md` を**セットで同期更新**する。
  - 設定・語彙・キャラ役割を変更する際は、変更前に grep 等で**全使用箇所を機械的に列挙**し、更新スコープを宣言してから「正本→派生」（2.1参照）の順で一括修正する。「検討」だけで済ませず、列挙という手順を必ず踏むこと（手順の曖昧さが片側更新・取り残しの原因になった実績がある）。
  - 決定によって語彙・概念を**廃止・置換**した場合は、grep全列挙とセットで死語パターンを `00_system/deprecated_terms.yaml` へ登録する。登録が無いと、後から残骸検査を機械的に打てなくなる（旧稿残骸の検出が目視頼みになった実績への対処）。
  - 下層（L2〜L4）で確定したメカニズム・語彙が、上層（L1 `concept.yaml` 等）の記述と矛盾していないか**還流チェック**を行う。還流は L4→L3（catharsis_core）に限らず、全階層間で双方向に行う。
- **[Hook: 成果物の生成前・ルール設計前]**
  - 新たなプロットや本文を生成する前、またはシステムルールを追加・変更する前には、必ず `00_system/SELF_REVIEW.md` を読み込み、反パターンに陥っていないか自己検証（セルフレビュー）を行うこと。§3はフェーズ別（§3a 設定・世界観／§3b プロット・構造・敵設計／§3c 執筆・レンダー）に分かれている——現フェーズに該当する節＋全フェーズ横断の §1・§2 を読む（節の対応は §3 冒頭のルーティング表）。
- **[Hook: L4執筆時（本文生成ワークフロー）]**
  - 標準フロー: **①未決列挙 → ②対話解決 → ③初版自動生成＋判断ログ → ④自動バリデーション → ⑤人間判定 → ⑥還流または差し戻し**。
  - **① 未決列挙・② 対話解決**: scene_input作成時、上位サマリーが決めていない事項を `open_questions` に列挙し、人間との対話（最小質問）で解決してから生成に入る。手形の解決は執筆中の場当たり補完でなく執筆前に行う。
  - **事象時系列（event_timeline）の確立**: scene_input に描写中立の原子的事象列（WHAT のみ＝動詞と事実）を順に置く。POV・文体・描写密度を変えた再生成は、この事象列を固定参照して再現度を確保する（語り非依存の正本）。物理整合・行動妥当性は**描写を盛る前にこの事象レベルで検算・修正**する（修正は全レンダーに一括継承される）。規律3点——①WHAT限定（情景の再話＝HOW を混ぜない。混ぜると arc_summary の episode_index と同じ構造的ドリフトを招く）②正本なのは起稿〜再生成フェーズのみ＝`07_final_text` 確定後は本文が正本に戻り event_timeline は履歴（双方向同期を永続させない）③採用発明は decision_log 経由で event_timeline へ還流し、章核に関わるものは `catharsis_core` へ上げる。
  - **③ 初版自動生成**: catharsis_coreゲート通過済み・open_questions全解決済みなら、初版は完全自動生成してよい（人間は白紙への質問よりドラフトへの違和感の方が答えやすい）。ただしサマリー・scene_inputに無い判断（解釈・補完・発明）は `decision_log` として本文と必ずセットで提出する——暗黙に埋まった判断はそれらしく読めてしまうため検出不能になる。
  - **③' レンダー層セルフパス（初稿提示前・必須）**: 生成AIは初稿を人間へ出す前に本文を自己照合する——(1)**前提整合**：各文が `scene_facts` / `event_timeline` と矛盾しないか（例：職探し中の主人公に『自分の仕事』と書く自己矛盾）(2)**決め台詞の予約**：`dialogue_samples` 等の signature を地の文の埋め草に流用していないか（決め台詞は話者に言わせる場でこそ効く）(3)**文の自己完結性**：締め文・指示語（『それだけのことだ』等）が単独で何を指すか曖昧でないか(4)**係り受け・コロケーションの自然さ**：音読相当の再読で各語の係りが自然か（例：『Aの続き／Aを探す』で活動を指すべき位置に場所・物が嵌まっていないか＝×斡旋所の続き→○引き続き斡旋所を）。(1)(2)は機械的な滑りでありここで潰す。(3)(4)の『曖昧・不自然な係り受け』は検証可能性バイアスの境界（書き手は成立と思い込む）ゆえ自己検出は完全でない——人間判定（⑤）をループ終了条件として残す。4点に限る（過剰チェックリスト化は質を痩せさせる＝episode_index・4幕の轍）。**自然さを有限パターンのgrepルールにしない**（意味依存ゆえ誤検出で破綻する。再発した特定誤用のみ tripwire を deprecated_terms 方式で足すのは可）。
  - **④ 自動バリデーション**: 機械検査可能な項目のみ（【負荷】上限・変数整合・スキーマ準拠・design_notesの消化漏れ）。検査器＝`00_system/validate.py`（2026-07-02新設・依存ゼロ）——scene_input／chapter_summary のスキーマ必須キー・open_questions全resolved（生成ゲート）・【負荷】≤80%・sense/empathy/peak_variable語彙・validation_hookフラグを検査する。design_notesの消化漏れは意味依存のため対象外＝レンダー層セルフパス（③'）と人間判定（⑤）の領分。
  - **⑤ 人間判定**: 「文章として実際に面白いか」は検証項目化できない（検証可能性バイアス）。自動ループだけ回すと全チェック通過の平板な文章が生まれるため、**人間の読了判定をループの終了条件**とする。人間がレビューするのは本文とdecision_logの両方。
  - **⑥ 還流**: 出来事レベルの発明・修正は人間承認後、まず scene_input の `event_timeline` へ還流し（再生成の再現度を保つため）、章核に関わるものは該当章の `catharsis_core` へ上げる。文章レベルの発明は還流不要。
  - **章完成時の台帳反映**: 人間判定を通過した章の decision_log・本文から、台帳対象（長距離の事実・累積カウンタ・認識/誤読）の変化のみを `02_dynamic_states/` へ反映する（2.1「本文と動的状態」・README §3参照）。
  - **文体アンカーの先行確立**: 作品の声が未確立な最初期（第1話前後）は、本格自動化の前に文体サンプルの対話的すり合わせ（A/B提示→選択）を一度挟む。最初の自動ドラフトが声を固定する（アンカリング）ため。声の確立後は自動化の比重を上げてよい。
- **[Hook: 成果物の格納時（合意形成後）]**
  - 設定ファイル等はAIが勝手に正式ディレクトリへ移動せず、ユーザーと壁打ちして「合意を得た」ものだけを移行・保存する。
- **[Hook: コミット前]**
  - **残骸の機械検査**: `00_system/deprecated_terms.yaml` の全パターンを実データ領域（01_static_database / 02_dynamic_states / 03_macro_concept / 04_meso_plot / 05_micro_scene / 06_draft_output / 07_final_text）へ grep し、ヒットを確認する。ヒット＝即違反ではなく文脈確認の対象（台帳の note 参照）。_workspace と 00_system は対象外。
  - **機械化済（2026-07-02）**: この残骸grepと CONTEXT/TODO の反映確認リマインドは、バージョン管理された git pre-commit フック `00_system/git_hooks/pre-commit` が全コミットで自動実行する（非ブロック表示）。AIが手動grepを再実行する必要はないが、**ヒットの文脈確認（新規ヒットが正当か）はフック出力を読んで行う**こと。新規クローン時の有効化: `git config core.hooksPath 00_system/git_hooks`。
  - `git commit` を実行する前に、会話中で得られた情報を以下の領域ごとに確認し、未反映があればコミット前に書き込む。
    - **作業状態領域** `_workspace/CONTEXT.md` `_workspace/TODO.md`
      → 確定事項・未決論点・次セッションへの引き継ぎ
    - **ルール領域** `00_system/ARCHITECTURE.md` `00_system/SELF_REVIEW.md` `00_system/variable_rules.md`
      → AIの振る舞い・運用フック・セルフレビュー項目・変数ルールの追加・変更
    - **静的DBおよびプロット領域** `01_static_database/` `04_meso_plot/`
      → 設定・語彙・プロットの確定事項（通常は変更ファイルとしてコミットに含まれているはず）
  - 「純粋な修正のみ」のコミットでも、その変更に至った背景にルール更新・引き継ぎが必要な決定事項が含まれていないか確認する。
- **[Hook: セッション終了・タスク質変化時]**
  - タスクの質が変わる際（整理→執筆など）はリセット（新規チャット）を提案する。
  - その提案の前に、必ずArtifacts等の一時領域にある草案を `_workspace/` に永続化（保存）してデータの揮発を防ぐ。
- **[Hook: 進行停滞・未完了TODO存在時]**
  - プロジェクトに未消化のTODOがある場合、執筆効率を最大化する観点で「最も効率的な情報整理・壁打ち（不要なノイズの削ぎ落としや最短ルートの提案）」を提案する。
